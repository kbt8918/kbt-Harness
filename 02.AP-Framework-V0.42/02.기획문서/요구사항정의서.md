# 요구사항 정의서 (Requirements Definition)

**프로젝트명**: 부모님 위치 확인 서비스
**작성일**: 2026-06-09
**버전**: v1.0
**작성자**: kbt8918 (기획자)
**근거 자료**: 서비스기획서.md(v1.0), 착수보고서.md, PRD.md

> 본 문서는 서비스기획서의 핵심 기능 5종·3개 사용자 그룹·차별화 전략을 요구사항으로 구조화한다.
> 본 문서의 결과는 기능명세서(#6)의 입력 자료가 된다.
> 사용자 구분 약어: P(부모님/위치 제공자) / F(가족/위치 확인 회원) / A(관리자/운영자) / C(공통)

---

## 1. 기능 요구사항 (Functional Requirements)

### 1.1 부모님 화면 (위치 제공자)

| ID | 구분 | 요구사항 | 우선순위 | 인수 기준 |
|----|------|----------|----------|-----------|
| REQ-P-001 | 위치 전송 | 부모님 단말의 GPS(Geolocation) 위치를 주기적으로 수집·전송한다 | Must | 위치 권한 허용 시 좌표가 서버로 전송되고 가족 화면 지도에 반영됨 |
| REQ-P-002 | 긴급 알림 | 큰 긴급 버튼 1회 터치로 가족에게 즉시 알림을 보낸다 | Must | 버튼 1회 터치 → 가족에게 알림 발송, 추가 단계 없음 |
| REQ-P-003 | 발송 피드백 | 알림 발송 시 부모님이 인지할 수 있는 시각적 피드백을 제공한다 | Must | 발송 직후 "전송됨" 등 명확한 시각 표시 노출 |
| REQ-P-004 | 고령자 UI | 큰 버튼·큰 글씨·고대비의 단순 화면을 제공한다 | Must | 주요 동작 버튼 1개 중심, 텍스트 가독성 기준 충족 |
| REQ-P-005 | 안심 메시징 | "감시가 아닌 안심·연결" 톤의 문구를 적용한다 | Should | 감시 연상 문구 배제, 안심·연결 메시지 노출 |

### 1.2 가족 화면 (위치 확인 회원)

| ID | 구분 | 요구사항 | 우선순위 | 인수 기준 |
|----|------|----------|----------|-----------|
| REQ-F-001 | 위치 지도 | 부모님 실시간 위치를 지도에 표시한다 | Must | 지도 API에 부모님 현재 위치 마커 표시, 위치 갱신 반영 |
| REQ-F-002 | 실시간 갱신 | 위치·긴급 알림이 수 초 내 화면에 반영된다 | Must | 발송→수신 지연 수 초 이내(SSE/WebSocket/Realtime) |
| REQ-F-003 | 바로 전화 | 버튼 클릭 시 부모님에게 즉시 통화 연결한다 | Must | 버튼 클릭 → tel: 통화 실행 |
| REQ-F-004 | 가족 채팅 | 가족 구성원 간 실시간 채팅방을 제공한다 | Must | 메시지 송수신 실시간 반영, 가족 단위 채팅방 |
| REQ-F-005 | 긴급 알림 수신 | 부모님 긴급 알림을 눈에 띄게 수신·표시한다 | Must | 긴급 알림 수신 시 일반 알림과 구분되는 강조 표시 |

### 1.3 관리자 화면 (운영자)

| ID | 구분 | 요구사항 | 우선순위 | 인수 기준 |
|----|------|----------|----------|-----------|
| REQ-A-001 | 회원 조회 | 연동 회원(부모-가족 매핑) 정보를 조회한다 | Must | 회원 목록·매핑 관계 조회 가능 |
| REQ-A-002 | SMS 발송 | 회원에게 SMS 다이렉트 발송한다 | Must | 선택 회원에게 SMS 발송 성공 |
| REQ-A-003 | 카카오톡 발송 | 회원에게 카카오톡 알림톡을 발송한다 | Should | 선택 회원에게 알림톡 발송 성공 |

### 1.4 공통 (인증·권한·매핑)

| ID | 구분 | 요구사항 | 우선순위 | 인수 기준 |
|----|------|----------|----------|-----------|
| REQ-C-001 | 권한 분리 | 부모님/가족/관리자 화면별 접근 권한을 분리한다 | Must | 각 역할은 자기 화면만 접근, 타 역할 화면 차단 |
| REQ-C-002 | 회원 매핑 | 부모님과 가족 계정을 사전 연동(매핑)한다 | Must | 매핑된 가족만 해당 부모님 위치·알림 확인 가능 |
| REQ-C-003 | 인증 | 가족·관리자 로그인 인증을 제공한다 | Must | 미인증 사용자 접근 차단 |
| REQ-C-004 | 반응형 | PC/태블릿/모바일 반응형 레이아웃을 제공한다 | Must | 주요 해상도에서 레이아웃 정상 표시 |

## 2. 비기능 요구사항 (Non-Functional Requirements)

| ID | 구분 | 요구사항 | 기준값 |
|----|------|----------|--------|
| NFR-001 | 성능(실시간) | 긴급 알림·위치 갱신 전달 지연 | 발송→가족 수신 수 초 이내 |
| NFR-002 | 보안 | 위치·개인정보 전송/저장 보호 | 전송 암호화, 화면별 권한 분리, service_role 프론트 노출 금지 |
| NFR-003 | 법규 준수 | 개인정보보호법·위치정보법 준수 | 위치정보 최소수집·동의 기반 수집, 보관 정책 정의(OI-04) |
| NFR-004 | 접근성 | 고령자 친화 가독성 | 큰 버튼·큰 글씨·고대비, 1-터치 조작(부모님 화면) |
| NFR-005 | 가용성 | 서비스 안정 운영 | Vercel 통합 배포 기준, 핵심 기능 5종 라이브 동작 |
| NFR-006 | 확장성 | 트래픽 증가 대응 | M1~M4 통합, 한도 초과 시 M5 Express 분리 |
| NFR-007 | 사용성 | 부모님 조작 단순성 | 긴급 알림 1-터치 달성, 사용성 테스트 통과 |
| NFR-008 | 배터리/데이터 | 단말 부담 최소화 | 위치 갱신 주기 최적화(OI-01), 향후 배터리 잔량 공유 |
| NFR-009 | 호환성 | 외부 API 연동 | 지도/SMS/카카오 사업자 키 확보(OI-05), 대체 후보 |

## 3. 우선순위 매트릭스 (MoSCoW)

| 우선순위 | 요구사항 ID | 비율 |
|----------|-------------|------|
| **Must** (필수) | REQ-P-001~004, REQ-F-001~005, REQ-A-001~002, REQ-C-001~004, NFR-001~007 | 약 80% |
| **Should** (중요) | REQ-P-005, REQ-A-003, NFR-008~009 | 약 15% |
| **Could** (선택) | 비활성·낙상 감지, 배터리 잔량 공유(확장 기능) | 약 5% |
| **Won't** (제외) | 네이티브 앱, 건강 모니터링(웨어러블), 결제/구독, 다국어, 위치 이력 분석 | (착수보고서 3.2 범위 외) |

## 4. 사용자 스토리

| ID | As a [사용자] | I want [기능] | So that [가치] |
|----|---------------|---------------|----------------|
| US-001 | 부모님 | 큰 버튼 1회만 누르고 싶다 | 복잡한 조작 없이 가족에게 위급을 알릴 수 있다 |
| US-002 | 부모님 | 내가 보낸 알림이 전송됐는지 보고 싶다 | 제대로 전달됐는지 안심할 수 있다 |
| US-003 | 부모님 | 감시당한다는 느낌 없이 쓰고 싶다 | 거부감 없이 서비스를 계속 사용한다 |
| US-004 | 가족 | 부모님 위치를 지도에서 실시간으로 보고 싶다 | 멀리 있어도 안부를 즉시 확인한다 |
| US-005 | 가족 | 긴급 알림을 즉시 받고 싶다 | 위급 상황에 빠르게 대응한다 |
| US-006 | 가족 | 버튼 한 번으로 부모님께 전화하고 싶다 | 번호를 찾지 않고 바로 통화한다 |
| US-007 | 가족 | 형제·가족과 한 채팅방에서 소통하고 싶다 | 대응을 함께 조율한다 |
| US-008 | 관리자 | 연동 회원을 조회하고 싶다 | 운영 대상을 한눈에 파악한다 |
| US-009 | 관리자 | 회원에게 SMS·카카오톡을 발송하고 싶다 | 공지·안내를 일괄·직접 전달한다 |
