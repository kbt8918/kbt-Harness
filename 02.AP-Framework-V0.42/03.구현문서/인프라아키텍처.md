> 본 문서는 화면설계서의 5개 화면(SCR-001~005)과 그 실시간/권한/반응형 요구를 운영하기 위한 인프라 구성을 정의한다.
> **현재 단계: M1~M4 통합** — Frontend + API를 Next.js(App Router) 단일 앱으로 빌드하여 **Vercel 통합 배포**한다. `src/backend/`(Express)는 빈 폴더로 유지하며, 트래픽 증가 시 **M5+에서 분리 호스팅**(Render/Railway)으로 전환한다.
>
> **[2026-06-09 변경] 관리자 백엔드 분리 선행 적용**: 사용자 결정에 따라 관리자 API(인증·회원조회·SMS/카카오 발송·매핑)를 `src/backend/`에 **독립 Express.js 서버**로 구현(M5+ 분리 전략을 선행). Supabase(PostgreSQL, service_role) 연동, JWT 인증, 역할 기반 접근 제어(F-012) 적용. 배포는 Vercel 별도 프로젝트(`kbt-harness-admin`) 또는 Render/Railway. 프론트엔드(가족/부모님 SCR-001~004)는 기존대로 Next.js 통합 유지. 상세는 `src/backend/README.md` 참조.

## 1. 시스템 구성도

```
                          [ 사용자 단말 ]
        ┌──────────────┬───────────────┬──────────────┐
        │  부모님(모바일) │  가족(PC/모바일) │  관리자(PC)    │
        │  SCR-002     │  SCR-003/004  │  SCR-005     │
        └──────┬───────┴───────┬───────┴──────┬───────┘
               │  HTTPS / SSE   │              │
               ▼                ▼              ▼
        ┌───────────────────────────────────────────────┐
        │            Vercel Edge Network (CDN)            │  정적 자산·이미지 캐시, TLS 종단
        └───────────────────────┬───────────────────────┘
                                 ▼
        ┌───────────────────────────────────────────────┐
        │     Next.js App (App Router) — Vercel 통합       │
        │  ┌─────────────────┐   ┌──────────────────────┐ │
        │  │ Frontend (SSR/CSR)│  │ API Route Handlers   │ │
        │  │ 화면 렌더·반응형   │   │ /api/* (API-001~015) │ │
        │  └─────────────────┘   └──────────┬───────────┘ │
        │                  authz 미들웨어(F-012) 공통 통과   │
        └───────┬──────────────────┬───────────────┬──────┘
                │                  │               │
                ▼                  ▼               ▼
     ┌──────────────────┐  ┌──────────────┐  ┌──────────────────┐
     │ Supabase          │  │ 실시간 채널    │  │ 외부 연동 서비스    │
     │ PostgreSQL(DB)    │  │ SSE /api/stream│ │ 지도 / SMS / 카카오 │
     │ 회원·매핑·위치·채팅 │  │ 또는 Realtime  │  │ Geolocation(단말)  │
     └──────────────────┘  └──────────────┘  └──────────────────┘
```

- **단일 앱 통합(M1~M4)**: 화면 렌더와 API가 동일 Next.js 앱에 존재하여 Vercel에 함께 배포된다.
- **실시간 경로**: 위치 갱신(F-004)·긴급 알림(F-005)·채팅(F-007)은 SSE(`/api/stream`) 또는 Supabase Realtime으로 가족 화면(SCR-003/004)에 전파된다.
- **외부 의존**: 지도 API(SCR-003), SMS·카카오 알림톡(SCR-005), 브라우저 Geolocation(SCR-002)은 외부/단말 자원에 의존한다.

## 2. 서버 구성

> M1~M4는 자체 서버를 운영하지 않고 Vercel(서버리스) + Supabase(매니지드 DB)에 위임한다. 아래는 논리 구성이다.

| 구분 | 환경 | 호스팅/사양 | 용도 |
|------|------|------------|------|
| Web/API (통합) | 개발/프리뷰/운영 | Vercel Serverless Functions (자동 스케일) | 화면 SSR/CSR + API Route Handlers(`/api/*`) |
| 실시간 채널 | 운영 | Vercel Function(SSE) 또는 Supabase Realtime | 위치·긴급·채팅 이벤트 브로드캐스트(F-002/005/007) |
| DB Server | 개발/운영 | Supabase PostgreSQL 15.x (매니지드) | 회원·매핑·위치·채팅·발송이력 저장 |
| Storage/CDN | 운영 | Vercel Edge Network | 정적 자산·이미지 캐시, 글로벌 배포 |
| (M5+ 예정) API 분리 | 운영 | Express.js / Render·Railway | 트래픽 증가 시 `src/backend/` 분리 호스팅 |

> SSE 장기 연결은 서버리스 함수 실행시간 제한을 받으므로, 연결 끊김 시 **자동 재연결 + 누락 이벤트 폴링 보정**(F-005)을 클라이언트에서 보장한다. 동시 실시간 부하가 커지면 Supabase Realtime(WebSocket) 또는 M5+ 분리로 전환한다.

## 3. 네트워크 구성

| 항목 | 내용 |
|------|------|
| 도메인 | 운영 `app.<도메인>` / 프리뷰 `*-git-<branch>.vercel.app`(자동) |
| SSL | Vercel 자동 발급·갱신(Let's Encrypt), 전 구간 HTTPS 강제(HSTS) |
| 방화벽/접근 | DB는 Supabase 측 접근 제어, `service_role` 키는 서버(API Route)에서만 사용·프론트 노출 금지(NFR-002) |
| 인바운드 | 443(HTTPS) 만 공개. 클라이언트 → Vercel → Supabase |
| 아웃바운드 | API Route → Supabase(DB/Realtime), → 지도/SMS/카카오 외부 API(HTTPS) |
| CORS | 동일 출처(통합 앱) 기본, 외부 도메인 호출은 허용 목록 관리 |

> 개인정보(위치·연락처)는 최소수집(OI-04) 및 전송 암호화(TLS)를 적용하고, 관리자 화면(SCR-005) 연락처는 마스킹 저장/표시한다.

## 4. 배포 환경

| 환경 | URL | 용도 | 배포 방식 |
|------|-----|------|-----------|
| 개발(Local) | `http://localhost:3000` | 개발/디버깅 | 로컬 실행(`next dev`) |
| 프리뷰(Staging) | `*-git-<branch>.vercel.app` | QA/테스트, PR 검토 | Git push 시 Vercel 자동 프리뷰 |
| 운영(Production) | `https://app.<도메인>` | 서비스 운영 | `main` 머지 시 GitHub Actions → Vercel 자동 배포(CI/CD) |

**CI/CD 파이프라인 (GitHub Actions)**

```
git push / PR
   → Lint(ESLint) + Build(next build) + Test(tests/)
   → Vercel 프리뷰 배포 (PR)
   → main 머지 시 운영 배포
```

> 배포 후 라이브 사이트 수동 검증 필수(Mock Pass ≠ 라이브 동작). Vercel 알려진 이슈 주의: `!` Body Escaping, bcrypt→bcryptjs, Supabase IPv6, `service_role` 프론트 노출 금지.

## 5. 환경 변수 / 키 관리

| 키 | 용도 | 사용 위치 | 관리 |
|----|------|----------|------|
| `DATABASE_URL` / Supabase URL·anon key | DB·Realtime 연결 | API Route / 클라이언트(anon) | Vercel 환경변수 + `.AP-key.md` |
| `SUPABASE_SERVICE_ROLE` | 서버 권한 작업 | API Route(서버) 전용 | 프론트 노출 금지(NFR-002) |
| `JWT_SECRET` | 토큰 서명·검증(F-011) | API Route(서버) | Vercel 환경변수 |
| 지도 API 키 | 지도 렌더(F-004, SCR-003) | 클라이언트(도메인 제한) | 키 사용 도메인 제한 |
| SMS / 카카오 알림톡 키 | 외부 발송(F-009/010) | API Route(서버) | 서버 전용 |

> 모든 서비스 키/URL은 `.AP-key.md`(.gitignore)와 Vercel 프로젝트 환경변수에 동시 관리한다.

## 6. 외부 연동 서비스

| 서비스 | 용도 | 관련 화면/기능 | 연동 방식 |
|--------|------|---------------|-----------|
| 지도 API (카카오/네이버/Google) | 부모님 위치 지도·마커 | SCR-003 / F-004 | JS SDK (클라이언트) |
| 브라우저 Geolocation API | 부모님 단말 위치 수집 | SCR-002 / F-001 | 브라우저 API (권한 동의·재요청) |
| SMS 사업자 API | 회원 SMS 발송 | SCR-005 / F-009 | REST API (서버, 재시도·결과 기록) |
| 카카오톡 알림톡 API | 승인 템플릿 발송 | SCR-005 / F-010 | REST API (서버, 미승인 템플릿 차단) |
| tel: (단말 통화) | 바로 전화 | SCR-003 / F-006 | `tel:` 링크 (단말 통화 앱) |
| Supabase Realtime / SSE | 위치·긴급·채팅 실시간 전파 | SCR-002→003, SCR-004 / F-002·005·007 | SSE(`/api/stream`) 또는 WebSocket |

> 외부 API 오류 시 재시도 후 사용자에게 명확한 실패 안내(F-009/F-010), 외부 키는 서버 측에서만 사용한다.

## 7. 비기능 요건 매핑 (인프라 관점)

| 요건 | 인프라 대응 |

| 운영 검증 | 프리뷰 환경 QA + 운영 배포 후 라이브 수동 검증 |
